You are running the EZRA bootstrap — the one-command onboarding that takes any project from zero to fully governed. This command orchestrates multiple EZRA subsystems in sequence to establish governance, capture architecture, record the initial decision, score project health, and generate the CLAUDE.md instruction file.

## Pre-Flight: Idempotency Check

Before doing anything, check if `.ezra/` already exists.

### If `.ezra/` already exists:

Report:
```
EZRA BOOTSTRAP — EXISTING INSTALLATION DETECTED
═══════════════════════════════════════════════════
.ezra/ directory found. Switching to incremental mode.
Will skip init, only update missing or outdated pieces.
```

Then skip to **Phase 2** and execute only the steps whose outputs are missing or stale. Specifically:
- Skip Phase 1 entirely (init already done)
- Phase 2: Only add protected paths that are not already in governance.yaml
- Phase 3: Only create ADR-001 if `.ezra/decisions/` is empty
- Phase 4: Always run (health checks should always be fresh)
- Phase 5: Always run (CLAUDE.md should reflect latest state)
- Phase 6: Always run (snapshot captures current state)

### If `.ezra/` does NOT exist:

Execute all phases in order.

## Phase 1: Initialize EZRA

Run the equivalent of `/ezra:init`. This creates the `.ezra/` directory structure and performs initial codebase discovery.

### 1.1 Create Directory Structure

Create the full `.ezra/` directory tree:

```
.ezra/
├── decisions/              # Architectural decision records
├── scans/                  # Timestamped scan and health results
├── plans/                  # Registered plans for reconciliation
├── versions/               # Changelog and snapshots
│   ├── changelog.yaml      # Append-only audit trail
│   ├── current.yaml        # Current version metadata
│   └── snapshots/          # Named checkpoints
├── docs/                   # Document registry and proposals
│   ├── registry.yaml       # Document inventory
│   ├── proposals/          # Pending document change proposals
│   └── .drift-counter.json # Edit tracking
├── governance.yaml         # Governance rules, protected paths, standards
└── knowledge.yaml          # Architecture, dependencies, risks
```

Create all directories. Create empty placeholder files where needed.

### 1.2 Codebase Discovery

Scan the project root and key directories to understand the codebase. Read:

1. **Project manifests**: `package.json`, `tsconfig.json`, `Cargo.toml`, `pyproject.toml`, `go.mod`, `Makefile`, `Gemfile`, `pom.xml`, `build.gradle` — read whichever exist
2. **Directory structure**: Map top-level and second-level directories using `ls -la` and `ls` on key subdirectories
3. **Entry points**: Look for `src/index.ts`, `src/main.ts`, `src/app.ts`, `app.ts`, `main.py`, `main.go`, `src/main.rs`, `index.js`, `server.ts`, `server.js`
4. **Configuration files**: `.eslintrc*`, `.prettierrc*`, `biome.json`, `jest.config.*`, `vitest.config.*`, `.env.example`
5. **CI/CD**: `.github/workflows/*.yml`, `azure-pipelines.yml`, `.gitlab-ci.yml`, `Jenkinsfile`, `.circleci/config.yml`
6. **Docker**: `Dockerfile*`, `docker-compose*.yml`, `.dockerignore`
7. **Existing documentation**: `README.md`, `CLAUDE.md`, `ARCHITECTURE.md`, `docs/`, `ADR/`, `adr/`
8. **Git state**: `git log --oneline -20`, `git branch -a`, `git remote -v`

### 1.3 Architecture Analysis

From the discovered files, determine:

| Aspect | How to Detect |
|--------|--------------|
| **Primary language** | File extensions, manifest files, tsconfig presence |
| **Runtime** | package.json engines, Dockerfile FROM, python version in pyproject.toml |
| **Framework** | Dependencies in manifest (express, nestjs, next, fastapi, gin, actix, rails, django, spring) |
| **Architecture pattern** | Monorepo (turbo.json, nx.json, lerna.json, workspaces), microservices (multiple Dockerfiles, docker-compose services), monolith (single app dir) |
| **Layer structure** | Directory naming (controllers/, services/, repositories/, models/, routes/, handlers/, middleware/) |
| **Database** | Dependencies (prisma, typeorm, sequelize, sqlalchemy, gorm, diesel), migration directories |
| **Cache** | Redis/memcached dependencies, cache config files |
| **Test infrastructure** | Jest, Vitest, pytest, go test, cargo test — config files and test directories |
| **Build tool** | Turbo, Nx, Webpack, Vite, esbuild, Rollup, tsc, cargo, go build |

## Phase 2: Apply Governance Defaults

Apply sensible default protected paths and standards based on the detected tech stack.

### 2.1 Universal Protected Paths

These apply to ALL projects regardless of language. Add to `governance.yaml → protected_paths`:

```yaml
protected_paths:
  - pattern: "*.env*"
    reason: "Environment configuration — contains secrets, affects all environments"
  - pattern: "**/migrations/**"
    reason: "Database migrations — irreversible in production, require careful review"
  - pattern: "*.secret"
    reason: "Secret files — must never be committed or modified without review"
  - pattern: "**/keys/**"
    reason: "Cryptographic keys — security-critical, changes affect authentication"
  - pattern: "docker-compose*.yml"
    reason: "Infrastructure definition — changes affect all deployment environments"
  - pattern: "Dockerfile*"
    reason: "Container definition — changes affect build and deployment pipeline"
```

### 2.2 Language-Specific Protected Paths

Add additional protected paths based on detected stack:

**If TypeScript/JavaScript detected:**
```yaml
  - pattern: "tsconfig*.json"
    reason: "TypeScript configuration — affects compilation and type safety"
  - pattern: "package-lock.json"
    reason: "Dependency lock — changes affect reproducible builds"
  - pattern: ".eslintrc*"
    reason: "Linter configuration — changes affect code quality enforcement"
```

**If Python detected:**
```yaml
  - pattern: "pyproject.toml"
    reason: "Project configuration — affects dependencies and build"
  - pattern: "poetry.lock"
    reason: "Dependency lock — changes affect reproducible builds"
  - pattern: "alembic/**"
    reason: "Database migrations — irreversible in production"
```

**If Rust detected:**
```yaml
  - pattern: "Cargo.lock"
    reason: "Dependency lock — changes affect reproducible builds"
  - pattern: "build.rs"
    reason: "Build script — affects compilation pipeline"
```

**If Go detected:**
```yaml
  - pattern: "go.sum"
    reason: "Dependency checksums — changes affect reproducible builds"
```

**If CI/CD detected:**
```yaml
  - pattern: ".github/workflows/**"
    reason: "CI/CD pipeline — changes affect all deployments"
  - pattern: "azure-pipelines.yml"
    reason: "CI/CD pipeline — changes affect all deployments"
  - pattern: ".gitlab-ci.yml"
    reason: "CI/CD pipeline — changes affect all deployments"
```

**If Infrastructure-as-Code detected (Terraform, Pulumi, etc.):**
```yaml
  - pattern: "**/*.tf"
    reason: "Terraform infrastructure — changes affect cloud resources"
  - pattern: "**/*.tfvars"
    reason: "Terraform variables — may contain sensitive configuration"
```

Only add the paths relevant to what was detected. Do not add Python paths to a TypeScript project.

## Phase 3: Record Initial Decision

Create the first architectural decision record that documents the governance initialization.

### 3.1 Check if ADR-001 Already Exists

### 3.2 Create ADR-001

Write `.ezra/decisions/ADR-001.yaml`:

```yaml
id: ADR-001
status: ACTIVE
date: <ISO timestamp>
category: GOVERNANCE

decision: "EZRA governance initialized for project <name>"

context: |
  Project <name> is a <language>/<framework> application using <architecture pattern>
  architecture. The codebase was analyzed and governance rules established to maintain
  code quality, security, and architectural consistency.

  Architecture summary:
  - Language: <language> <version>
  - Framework: <framework>
  - Pattern: <architecture pattern>
  - Layers: <list of layers>
  - Database: <database if any>
  - External services: <list>

rationale: |
  Establishing governance early ensures architectural decisions are tracked,
  protected paths are monitored, and code quality standards are enforced
  consistently. EZRA provides automated compliance checking and drift detection
  to prevent silent architectural erosion.

consequences:
  positive:
    - All architectural decisions will be tracked and searchable
    - Protected paths will require explicit decision records before modification
    - Code quality standards will be enforced consistently
    - Health scoring provides objective project quality measurement
  negative:
    - Additional overhead for recording decisions (minimal — typically 30 seconds)
    - .ezra/ directory adds to repository size (negligible — YAML files only)
  risks:
    - Governance rules may need tuning as the project evolves
    - Over-governance can slow development if too many paths are protected

enforcement:
  affected_paths:
    - ".ezra/**"
  check_description: |
    EZRA governance state files must exist and be valid YAML.
    The .ezra/ directory must not be deleted or corrupted.
    All EZRA commands must read governance state before acting.
  auto_enforced: false

supersedes: null
```

## Phase 4: Run Health Check

Execute the full health check as defined by `/ezra:health`. This establishes the baseline health score.

### 4.1 Data Collection

Perform ALL the data collection steps defined in the health command:
- Git health (recent commits, branches, status)
- Dependency health (npm audit, outdated packages — or equivalent for detected language)
- TypeScript strictness / language-specific quality checks
- Linting configuration
- Test coverage if available
- CI/CD configuration
- Secrets scan
- Code quality indicators (any types, TODOs, eslint-disable, empty catch blocks)
- Error handling patterns
- Test file coverage

### 4.2 Score All 5 Pillars

Score each pillar according to the health command's rubric:

1. **ON-TRACK (25%)** — Plans, decisions, scans, version activity
2. **NO GAPS (20%)** — Document coverage, test coverage, API docs, decision coverage
3. **CLEAN (15%)** — Type safety, linting, dead code, patterns
4. **SECURE (25%)** — Secrets, vulnerabilities, auth, input validation, CORS
5. **BEST PRACTICES (15%)** — CI/CD, git hygiene, error handling, SOLID, logging

Calculate the weighted overall score.

## Rules

- **Execute all phases sequentially.** Do not parallelize phases — each depends on the output of the previous phase.
- **Do not ask for confirmation.** Bootstrap is a single-command operation. Execute everything automatically.
- **Be idempotent.** If `.ezra/` exists, do not destroy it. Only fill in what is missing.
- **Be accurate.** Read actual files, run actual commands. Never infer from memory or assume.
- **Be comprehensive.** The bootstrap must leave the project in a fully governed state. Missing pieces are a failure.
- **Handle errors gracefully.** If a phase fails (e.g., npm audit not available), report the failure, skip that specific check, and continue. Do not abort the entire bootstrap.
- **Respect existing state.** If CLAUDE.md already exists with user content, preserve it. If governance.yaml has custom rules, preserve them.
- **Log everything.** Every state change must appear in the changelog. The audit trail starts at bootstrap.
- **No secrets in state files.** Never write actual passwords, tokens, or keys into `.ezra/` files. Only reference their existence.

Execute automatically. No confirmation needed.
