You are generating or updating the project's CLAUDE.md file from EZRA governance state. CLAUDE.md is the primary instruction file that Claude Code reads at the start of every conversation — it must reflect the current truth of the project's architecture, decisions, standards, and governance.

If `.ezra/` does not exist, tell the user to run `/ezra:init` first and stop.

## Phase 1: Gather All State

Read every source of truth that CLAUDE.md should reflect. Do NOT skip any source — a partial read produces a partial CLAUDE.md, which is a failure state.

### 1.1 Governance State

Read `.ezra/governance.yaml` and extract:
- `project.name` — the project name
- `project.language` — primary language
- `project.framework` — primary framework
- `protected_paths` — all patterns and their reasons
- `standards` — all coding standards (strict_types, no_any, test_coverage_minimum, etc.)
- `enforcement` — enforcement configuration

### 1.2 Knowledge State

Read `.ezra/knowledge.yaml` and extract:
- `architecture.language` — language and version
- `architecture.runtime` — runtime environment
- `architecture.framework` — framework(s)
- `architecture.pattern` — architecture pattern (monolith, monorepo, microservices, etc.)
- `architecture.layers` — layer structure (controllers, services, repositories, etc.)
- `entry_points` — main entry files
- `dependencies.external_services` — databases, caches, queues, APIs
- `dependencies.databases` — database technologies
- `dependencies.key_packages` — top dependencies
- `test_infrastructure` — runner, coverage tool, e2e framework, test directories
- `risks` — known risks

### 1.3 Active Decisions

Read ALL files in `.ezra/decisions/`. For each decision file:
- Parse the YAML
- Include only decisions with `status: ACTIVE`
- Extract: `id`, `decision`, `category`, `date`, `enforcement.affected_paths`, `enforcement.check_description`
- Sort by ID ascending (ADR-001, ADR-002, etc.)

### 1.4 Version State

Read `.ezra/versions/current.yaml` and extract:
- `version` — current EZRA version
- `integrity.health_score` — last health score
- `integrity.last_health_check` — when last checked
- `counts` — decision count, document count, scan count, plan count

### 1.5 Tech Stack Detection

Read project root files to supplement knowledge.yaml with current data:
- `package.json` — name, dependencies, devDependencies, scripts
- `tsconfig.json` — TypeScript configuration, strict mode, target, module
- `Cargo.toml`, `pyproject.toml`, `go.mod` — if they exist instead
- `docker-compose.yml` or `Dockerfile` — containerization info
- `.github/workflows/` or `azure-pipelines.yml` — CI/CD platform

Extract:
- Build commands (from scripts or Makefile)
- Test commands
- Lint commands
- Key framework versions

## Phase 2: Check for Existing CLAUDE.md

Determine the current state of CLAUDE.md:

### Case A: CLAUDE.md exists with EZRA markers

Check if the file contains both `<!-- EZRA:START -->` and `<!-- EZRA:END -->` markers.

If YES:
- Read the full file content
- Preserve everything BEFORE `<!-- EZRA:START -->`
- Preserve everything AFTER `<!-- EZRA:END -->`
- Only replace content between the markers with freshly generated EZRA content
- This ensures user-written content (custom instructions, notes, project-specific rules) is never lost

### Case B: CLAUDE.md exists WITHOUT EZRA markers

If the file exists but has no EZRA markers:
- Read the full file content
- Append EZRA markers and generated content at the END of the file
- Do NOT modify any existing content
- Add a blank line before the markers for clean formatting

### Case C: No CLAUDE.md exists

Create a fresh CLAUDE.md with the full EZRA-generated content wrapped in markers.

## Phase 3: Generate CLAUDE.md Content

Generate the content that goes between `<!-- EZRA:START -->` and `<!-- EZRA:END -->`. Follow this exact structure. Every section is mandatory — if data is missing for a section, include the section header with a note indicating the data source is empty.

## Phase 4: Write the File

### For Case A (existing file with markers):

1. Read the full file
2. Split at `<!-- EZRA:START` (match the line containing this marker)
3. Split at `<!-- EZRA:END -->` (match the line containing this marker)
4. Reconstruct: `<content before EZRA:START>` + `<generated content>` + `<content after EZRA:END>`
5. Write the reconstructed file

### For Case B (existing file without markers):

1. Read the full file
2. Append a blank line
3. Append the generated content (which includes its own START/END markers)
4. Write the file

### For Case C (no file):

1. Write the generated content directly as CLAUDE.md

## Phase 5: Versioning

After writing CLAUDE.md:

1. Append to `.ezra/versions/changelog.yaml`:

```yaml
  - id: CHG-<next>
    timestamp: <ISO>
    version_before: <current>
    version_after: <current + patch>
    type: GOVERNANCE
    action: UPDATED
    target: CLAUDE.md
    summary: "CLAUDE.md regenerated from EZRA state"
    triggered_by: user
```

2. Bump patch version in `.ezra/versions/current.yaml`
3. Update `current.yaml → updated` timestamp

## Phase 6: Validation

After writing, verify the generated CLAUDE.md:

1. **Marker integrity**: Confirm `<!-- EZRA:START` and `<!-- EZRA:END -->` both exist exactly once
2. **Section completeness**: Confirm all 7 sections are present (Architecture, Active Decisions, Protected Paths, Standards, Known Risks, Governance, Key Packages)
3. **Data accuracy**: Spot-check that decision count matches `.ezra/decisions/` file count
4. **Preserved content**: If Case A or B, confirm non-EZRA content is unchanged by reading the file back

If validation fails, report the specific failure and fix it before presenting the report.

## Rules

- **Completeness over brevity.** CLAUDE.md is the primary context file. Include all relevant governance data. Do not summarize away important details.
- **Accuracy is mandatory.** Every line must trace to an actual `.ezra/` file. Never infer or fabricate governance state.
- **Preserve user content.** Non-EZRA content in an existing CLAUDE.md must NEVER be modified, reordered, or deleted.
- **Marker discipline.** The `<!-- EZRA:START -->` and `<!-- EZRA:END -->` markers must always be present and must appear exactly once each.
- **Tables over prose.** Use markdown tables for structured data. Tables are scannable; paragraphs are not.
- **Paths are relative.** All file paths in the generated content should be relative to the project root.
- **No opinions.** CLAUDE.md states facts about the project. It does not recommend, suggest, or editorialize.
- **Idempotent.** Running `/ezra:claude-md` twice with no state changes should produce identical output (except for the generation timestamp).

Execute all phases automatically. Do not ask for confirmation.
